## 1. Ce qui marche vraiment (mesurable, vérifié dans le code)

### Couche socle — solide
- **Symfony 7.4 / PHP ≥ 8.2** propre, dépendances cohérentes (cf. composer.json).
- **3 entités Core** finies et bien écrites : `User`, `Club`, `UserClubRole`.
  - Lifecycle callbacks pour timestamps (`createdAt`, `updatedAt`).
  - Validation Symfony Assert sur tous les champs sensibles.
  - `UniqueEntity` sur email user + slug club.
  - Contrainte DB `UNIQUE(user_id, club_id, role)` côté pivot.
- **Consentement RGPD** explicite avec timestamp (`rgpdConsent` + `rgpdConsentAt`) — point fort jury.

### Sécurité — bien pensée
- **7 firewalls** isolés par host (dev, api, vitrine_admin, vitrine, manager, pirb, main).
- **Role hierarchy** : SUPER_ADMIN → DIRIGEANT → COACH → USER.
- **CSRF activé** sur tous les login forms.
- **ClubVoter** (5 attributs : MEMBER, COACH, ADMIN, STAFF, JOUEUR) + **TenantResolver** propre (session `active_club_id`, vérification d'appartenance, multi-clubs supporté).
- Séparation rôles Symfony / rôles métier (UserClubRole) bien pensée.

### Vitrine — fonctionnelle en prod
- 10+ routes publiques opérationnelles (accueil, news, club, équipes, équipes 3x3, galerie, calendrier, numérique, contact, victoires, article par slug).
- Pagination articles (`findPubliesPagines`).
- Formulaire contact avec validation + CSRF + mailer (mais voir alerte plus bas).
- 33 templates Twig.

### Admin CMS — utilisable
- CRUD complet Articles, Pages, Médias, Roles (6 controllers admin).
- Upload images avec Slugger.
- Éditeur WYSIWYG Quill 1.3.6 (après itération TinyMCE → Quill, choix justifié).
- 13 pages CMS éditables via back-office.

### Documentation — niveau au-dessus de la moyenne CDA
- 14 fichiers gouvernance (`00_GOUVERNANCE_DOC.md` → `14_SUIVI_CDC_MARS.md`).
- **6 ADR** documentées (architecture monolithe modulaire, host firewall, multi-tenant, hybride API+Twig, choix Symfony 7.4, RBAC).
- Registre sécurité RGPD (8 fiches RGPD-0001 à RGPD-0008).
- Dictionnaire DB de 115 KB.
- Definition of Done structurée par module.
- Claude log de 88 KB avec ~41 sessions tracées.
- **Diagramme PUML auto-généré** depuis les entités (`PumlCommand`).

### Infrastructure dev
- Docker Compose avec PostgreSQL 16 + Mailpit.
- Doctrine migrations (6 migrations versionnées).
- 3 sets de fixtures (User+Club, Articles, PageContenu).
- Healthcheck sur le service database.

## 2. Ce qui pue — sans filtre

### Bombe à retardement #1 : Stack DB incohérente
- `compose.yaml` → PostgreSQL 16-alpine
- `doctrine.yaml` → `PostgreSQLPlatform: identity`
- `.env` (ligne 6) → `DATABASE_URL="mysql://mabb_app:CHANGE_ME@127.0.0.1:3306/mabb_pirb..."`
- Migrations utilisent `AUTO_INCREMENT` + `TINYINT` (syntaxe MySQL)
- Page Notion "Récap Code & Soutenance" dit MySQL 8.4 sur OVH

**Tu ne sais pas toi-même quelle DB tu utilises.** Le jury va te bousculer là-dessus. Décide MySQL OU Postgres et purge l'autre — y compris en virant le `compose.yaml` Postgres si tu pars MySQL. ADR à créer.

### Bombe à retardement #2 : Zéro test
- `tests/` contient `bootstrap.php` et c'est tout.
- `phpunit.dist.xml` est en place mais aucun test n'a été écrit.
- Le référentiel CDA Bloc 1 (Développer une application sécurisée) **exige** des tests unitaires.
- Ta DoD section "Tests" est cochée "Phase 6, applicable rétroactivement" — c'est-à-dire jamais.

Sans 5-10 tests minimum sur `ClubVoter`, `TenantResolver`, et `UserClubRole::setRole()`, tu perds des points secs au jury.

### Bombe à retardement #3 : MABB **Manager** n'a pas de feature manager
Le nom du projet c'est "MABB Manager" mais :
- `src/Entity/Sport/` → **vide**
- `src/Entity/Pirb/` → **vide**
- `src/Controller/Manager/` → uniquement `ManagerLoginController` (login + stub dashboard avec badge "Sprint 2 en cours")
- `src/Controller/Pirb/` → uniquement `PirbLoginController` (idem)

Concrètement aujourd'hui, ton projet c'est : un **site vitrine + CMS** avec une auth multi-tenant pré-câblée pour un manager qui n'existe pas. Si le jury demande "montre-moi une feature métier de gestion d'équipe", tu n'as rien à montrer.

### Bombe à retardement #4 : API totalement absente
- `src/Controller/Api/` → `.gitkeep` uniquement.
- Firewall `api` configuré avec `jwt: ~` commenté.
- `LexikJWTAuthenticationBundle` cité dans 4 docs… jamais installé.
- ADR-0004 acte API Platform + JWT, suspended depuis Phase 1.

Aucune API REST en place. Si tu vises mobile V3 (cf. ADR-0004), c'est plus que mort.

### Bombe à retardement #5 : Deux systèmes de rôles parallèles
Dans `User.php` tu as :
- `array $rolesMembre = ['benevole']` (rôles vitrine — JSON column)
- `Collection $userClubRoles` (rôles métier multi-clubs)

Cohabitation jamais justifiée dans une ADR. Le jury va demander "à quoi sert le second ?", et tu vas hésiter. Choisis-en un et tue l'autre, ou documente clairement leur rôle distinct.

### Risque opérationnel : 9 commits non poussés (session 38)
Le Claude log du 26 mars mentionne "9 commits en attente de push". Si la machine crame, tu perds 1 session de travail. Vérifie `git status` et push.

### Risque opérationnel : Mailer null
- `.env` → `MAILER_DSN=null://null` (aucun mail envoyé)
- Le formulaire contact mabb.fr affiche "envoyé" mais c'est du bluff.
- Sans `.env.local` SMTP OVH côté serveur, la fonctionnalité contact est cassée en prod.

### Détails — pas urgents mais à noter
- `AccueilController::contact` concatène du HTML en string pour le mail (devrait être un template Twig email).
- `strip_tags` puis injection directe dans email body — pas critique mais peu propre.
- Pas de validation MIME/type sur les uploads (`AdminArticlesController::new`) — risque upload malveillant.
- `AdminUploadController` (TinyMCE — désormais Quill) reste exposé alors que Quill ne l'utilise plus → mort-né à supprimer.
- Pas de **README.md** à la racine du projet. Le jury va galérer à comprendre comment lancer.
- `rate_limiter` Symfony pas configuré sur les logins → bruteforce facile (cf. RGPD-0001 backlog).
- Stagnation : dernier commit le 26 mars 2026, soit ~7 semaines de pause au moment de l'audit.

## 5. Killshot questions — ce que le jury va te poser

Préparation **obligatoire** sur ces 12 questions :

1. *"Pourquoi 7 firewalls dans `security.yaml` ?"* → isolation par host + chaque espace a son provider et ses redirections distinctes.
2. *"Différence entre rôle Symfony et rôle métier ?"* → Symfony = ROLE_SUPER_ADMIN, ROLE_USER (plateforme). Métier = COACH, JOUEUR, BENEVOLE (UserClubRole, par club). Symfony gère l'accès aux firewalls/URLs, le métier gère l'accès aux ressources.
3. *"Comment tu garantis qu'un coach du club A ne voit pas les données du club B ?"* → `ClubVoter` + `TenantResolver` + chaque entité métier porte un `club_id`. Filtre obligatoire dans les repositories.
4. *"Qu'est-ce qu'un Voter Symfony et pourquoi utiliser ça plutôt que `access_control` ?"* → cf. déjà rédigé dans la page "Récap Code & Soutenance".
5. *"Pourquoi pas de tests aujourd'hui ?"* → réponse honnête : "Phase 1 socle, tests prévus en Phase 6. Je sais que c'est une faiblesse, voici les 5 tests que j'écris dans le mois qui vient."
6. *"Comment tu gères les migrations BDD ?"* → `doctrine:migrations:diff` génère depuis les attributs PHP, `doctrine:migrations:migrate` applique. Chaque migration est versionnée et a un `up()` et `down()`.
7. *"Tu utilises MySQL ou PostgreSQL ?"* → à trancher AVANT d'aller voir le jury.
8. *"Le multi-tenant, c'est quoi exactement ?"* → cf. déjà rédigé.
9. *"Comment tu protèges contre les injections SQL ?"* → Doctrine ORM utilise des requêtes préparées en interne. Pas de SQL brut dans le code applicatif.
10. *"Comment tu hashes les mots de passe ?"* → `password_hashers: 'auto'` dans `security.yaml` → Symfony choisit bcrypt ou argon2id selon l'environnement. Hash one-way, vérification via `password_verify`.
11. *"RGPD : montre-moi un endroit concret où tu l'as appliqué."* → `User::rgpdConsent` + `rgpdConsentAt` horodatés + `RGPD-0001` à `RGPD-0008` dans le registre + soft delete via `isActive` + droit à l'oubli prévu.
12. *"Si tu devais recommencer, qu'est-ce que tu ferais différemment ?"* → réponse à préparer (jury ADORE cette question). Honnête : commencer par la Phase 2 Sport plus tôt, écrire les tests dès la première entité, et ne pas avoir attendu Phase 6 pour les fondamentaux.
